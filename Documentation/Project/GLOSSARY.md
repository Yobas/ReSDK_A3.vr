# Глоссарий терминов проекта

Глоссарий содержит определения терминов, понятий и аббревиатур, используемых в проекте ReSDK_A3.

## Основные концепции

### Платформа
Игровой движок Arma 3 (Real Virtuality Engine), на котором построен проект ReSDK_A3. В документации используется термин "Платформа" для обозначения базовой системы, предоставляющей базовые типы данных, функции и возможности.

**См. также:** [Обзор проекта](01_OVERVIEW.md)

### OOP (Object-Oriented Programming)
Система объектно-ориентированного программирования для SQF, реализованная через макросы. Использует location объекты Платформы как основу для объектов, поддерживает наследование, методы и поля классов. Полностью описана в заголовочном файле `host/oop.hpp`.

**См. также:** [Особенности синтаксиса](03_SYNTAX_GUIDE.md), [Структура проекта](02_PROJECT_STRUCTURE.md)

### NOEngine (Net Objects Engine)
Система оптимизированной загрузки и синхронизации игровых объектов. Обеспечивает динамическую подгрузку частей карты (чанков) для каждого клиента, локальное создание объектов только в области видимости и программную репликацию состояния. Позволяет поддерживать тысячи объектов на карте без критичной потери производительности.

**См. также:** [Обзор проекта](01_OVERVIEW.md)

### RPC (Remote Procedure Call)
Система удаленных вызовов процедур для взаимодействия между клиентом и сервером. Позволяет клиенту вызывать функции на сервере и наоборот, обеспечивая синхронизацию состояния игрового мира.

**См. также:** [Особенности синтаксиса](03_SYNTAX_GUIDE.md)

### Структуры (Structures)
Система типизированных структур данных на основе HashMap объектов Платформы. Быстрее, чем location объекты OOP системы, но менее функциональны. Используется в NetDisplays, Overlays, AI Brain, CraftSystem.

**См. также:** [Особенности синтаксиса](03_SYNTAX_GUIDE.md), [Структура проекта](02_PROJECT_STRUCTURE.md)

## Архитектурные термины

### Host (Сервер)
Центральный узел, управляющий игровой логикой, обрабатывающий действия игроков и синхронизирующий состояние игрового мира. Все критичные операции выполняются на сервере.

**Код:** `Src/host/`

**См. также:** [Обзор проекта](01_OVERVIEW.md)

### Client (Клиент)
Локальное приложение игрока, отображающее игровой контент и отправляющее действия на сервер. Не выполняет критичную игровую логику.

**Код:** `Src/client/`

**См. также:** [Обзор проекта](01_OVERVIEW.md)

### Editor (Редактор)
Редактор ReEditor для создания и редактирования игрового контента. Использует специальную компонентную систему инициализации.

**Код:** `Src/Editor/`

**См. также:** [Структура проекта](02_PROJECT_STRUCTURE.md)

### Модули
Независимые единицы функциональности со своей точкой входа (`*_init.sqf`) и четко определенными зависимостями. Проект организован как модульная система с разделением на серверные, клиентские, Редакторные модули и общие компоненты.

**См. также:** [Модульная система](04_MODULE_SYSTEM.md)

### CommonComponents (Общие компоненты)
Модули в директории `host/CommonComponents/`, которые используются как на сервере, так и на клиенте. Содержат общие структуры данных, алгоритмы, математические функции, константы.

**Код:** `Src/host/CommonComponents/`

**См. также:** [Структура проекта](02_PROJECT_STRUCTURE.md)

### Зависимости модулей
Связи между модулями, определяющие порядок их загрузки. Критично загружать модули в правильном порядке из-за зависимостей.

**См. также:** [Модульная система](04_MODULE_SYSTEM.md), [Структура проекта](02_PROJECT_STRUCTURE.md)

## Технические термины

### Чанки (Chunks)
Части карты, на которые разделяется игровой мир в NOEngine. Загружаются динамически для каждого клиента в зависимости от его позиции, что позволяет оптимизировать сетевой трафик и производительность.

**См. также:** [Обзор проекта](01_OVERVIEW.md)

### null (nil)
Специальное значение в SQF, которое представляет отсутствие значения. Имеет особое поведение:
- В условных операторах `if (null) then {} else {}` обе ветки никогда не выполняются
- Операция `_array pushBack null` не добавит null в массив (нужно использовать `_array append [null]`)
- Передача null в функции может привести к пропуску операции
- Для проверки на null используются макросы: `isNullVar()`, `isNullReference()`, `valid()`

**См. также:** [Работа с null значениями](03_SYNTAX_GUIDE.md#работа-с-null-nil-значениями)

### Репликация
Процесс синхронизации состояния объектов между сервером и клиентами. В NOEngine репликация происходит программно, а не через стандартный механизм Платформы.

**См. также:** [Обзор проекта](01_OVERVIEW.md)

### Препроцессинг
Обработка кода перед выполнением: обработка макросов, `#include`, условная компиляция. Код проходит через препроцессор и затем загружается во время запуска игры/редактора.

**См. также:** [Обзор проекта](01_OVERVIEW.md)

### Компиляция
Формально процесс загрузки модулей называется "компиляцией", но фактически это препроцессинг и исполнение скриптов (которые загружаются в байт-код движком Платформы).

**См. также:** [Обзор проекта](01_OVERVIEW.md)

### Условная компиляция
Использование флагов (`DEBUG`, `RELEASE`, `EDITOR`, `RBUILDER`) для условного включения/исключения кода на этапе препроцессинга.

**См. также:** [Особенности синтаксиса](03_SYNTAX_GUIDE.md)

## Термины процессов

### Релиз
Официальная версия проекта, выпускаемая после тестирования и подготовки. Релизы автоматически собираются сервисом обновления при публикации на GitHub.

**См. также:** [Версионирование](07_VERSIONING.md)

### Сборка (Build)
Процесс преобразования исходного кода в исполняемые файлы для развертывания. Осуществляется через RBuilder для SQF кода и CMake для RVEngine.

**См. также:** [Сборка и деплой](06_BUILD_DEPLOY.md)

### Деплой (Deploy)
Процесс развертывания собранного проекта на серверах или клиентах. Выполняется автоматически сервисом обновления или вручную через RBuilder.

**См. также:** [Сборка и деплой](06_BUILD_DEPLOY.md)

### Версионирование
Система управления версиями проекта. Используется семантическое версионирование (MAJOR.MINOR.PATCH). Версия хранится в файле `Src/VERSION`.

**См. также:** [Версионирование](07_VERSIONING.md)

### Сервис обновления
Автоматическая система, которая отслеживает коммиты и релизы в репозитории GitHub, собирает релизные версии и развертывает их на серверах.

**См. также:** [Версионирование](07_VERSIONING.md), [Сборка и деплой](06_BUILD_DEPLOY.md)

### Симуляция
Запуск клиента и сервера из под редактора для тестирования функциональности. В режиме симуляции одновременно работают и серверная, и клиентская части кода, что позволяет тестировать сетевые взаимодействия и полный игровой цикл без необходимости запуска отдельных сервера и клиента.

**См. также:** [Отладка и диагностика](05_DEBUGGING.md)

## Аббревиатуры

### SQF
Scripting Language for Arma 3 - язык программирования, предоставляемый Платформой, используемый в проекте для написания игровой логики.

### PBO
Pack Binary Object - формат упакованных файлов, используемый Платформой для модов и миссий.

### RBuilder
Система сборки и деплоя SQF кода проекта. Обрабатывает препроцессинг и упаковывает код в форматы, необходимые для Платформы.

**См. также:** [Сборка и деплой](06_BUILD_DEPLOY.md)

### RVEngine
C++ расширение для Платформы, внешний компонент проекта с собственной системой сборки и документацией.

**См. также:** [Сборка и деплой](06_BUILD_DEPLOY.md)

### ReEditor
Редактор игрового контента проекта, основанный на редакторе Платформы (3DEN).

### ReNode
Система визуального программирования через графы узлов. Используется для создания игровой логики без написания кода.

### CBA
Community Based Addons - библиотека модов для Платформы, используется в проекте для различных утилит.

### API
Application Programming Interface - интерфейс программирования приложений, набор функций и методов для взаимодействия с модулями проекта.

**См. также:** [API документация](../API/README.md)

### PR
Pull Request - запрос на слияние изменений в основную ветку репозитория.

**См. также:** [Процессы разработки](08_DEVELOPMENT_WORKFLOW.md)

### CI/CD
Continuous Integration / Continuous Deployment - практика автоматической интеграции и развертывания кода.

### Git
Система контроля версий, используемая для управления исходным кодом проекта.

**См. также:** [Процессы разработки](08_DEVELOPMENT_WORKFLOW.md)

## Типы игровых объектов

### Item
Категория игровых объектов - предметы. Используются в NOEngine с определенной дистанцией прорисовки.

### IStruct
Категория игровых объектов - структуры. Используются в NOEngine с определенной дистанцией прорисовки.

### Decor
Категория игровых объектов - декоративные объекты. Используются в NOEngine с определенной дистанцией прорисовки.

### BasicMob
Категория игровых объектов - базовые мобы (неигровые персонажи). Используются в NOEngine с определенной дистанцией прорисовки.
